# ADR-002: 에러 대시보드 내 HTTP 폴링 방식 채택

> 작성일: 2026-07-11  
> 상태: 승인됨 (Accepted)

## 컨텍스트

대기열 관리 시스템(Rusui)의 본사 관리자 백오피스에서, 서버 내에서 발생하는 다양한 에러(4xx/5xx, DB 오류, SSE 연결 끊김)를 실시간으로 수집하고 보여주는 에러 대시보드 기능을 구축해야 합니다.  
기존 대기열의 실시간 고객 알림 기능에는 **SSE (Server-Sent Events)** 영속 연결 방식이 사용되고 있었으나, 본 에러 감시 대시보드 화면에서는 마찬가지로 SSE/WebSocket 같은 영속형 스트리밍을 채택할지, 아니면 주기적인 **HTTP 폴링 (Polling)** 방식을 적용할지에 대한 의사결정이 필요했습니다.

---

## 결정: 5초 주기의 HTTP 폴링 채택

에러 감시 시스템이 지녀야 할 최우선의 가치인 단순성(Simplicity), 그리고 최악의 서버 과부하 상태에서도 살아남을 수 있는 복원력(Resiliency)을 확보하기 위해, 에러 대시보드 데이터 수집 방식으로 **5초 주기 단기 HTTP 폴링(Polling) 기법**을 채택합니다.

### 채택 이유

1. **실시간성 요구수준의 상이성**:
   - 대기열 고객용 화면(SSE)은 입장 시점과 직결되어 있어 '1초 미만의 실시간 알림'이 강제되지만, 관리자용 모니터링 대시보드는 전반적인 장애 추이 관측이 목적이므로 약 5초 수준의 갱신 지연은 사용자 경험 및 실용성 측면에서 전혀 문제가 되지 않습니다.
2. **연결 유지 오버헤드 방지 및 자원 최소화**:
   - SSE와 같은 지속 연결 방식은 관리자 브라우저가 화면을 띄워놓고 있는 동안 지속적으로 서버 소켓 및 고루틴(Goroutine) 자원을 묶어두게 됩니다.
   - 반면 폴링 방식은 5초 간격으로 찰나의 요청을 보내고 바로 끊으므로 상시 점유 리소스 비용이 0에 수렴합니다. 대시보드 화면을 나가거나 브라우저 탭을 끄면 자원이 즉시 정리됩니다.
3. **서버 장애 상태에서의 높은 복원력(Resiliency)**:
   - 로깅/모니터링 인프라는 메인 서비스가 중단되거나 심각한 대역폭 포화를 겪고 있는 장애 상황에서 가장 견고하게 동작해야 합니다.
   - 영속 스트림 연결은 회선 불량 상황 시 복잡한 재연결 및 백오프(Backoff) 관리가 요구되지만, HTTP 폴링 방식은 표준 REST 통신을 사용하여 "실패 시 다음 5초 후에 그냥 다시 요청한다"는 아주 단순한 구조로 동작하므로 복원력이 매우 높습니다.

---

## 영향 (Consequences)

- **장점**:
  - 서버의 활성 소켓 유지 부하를 대폭 덜어낼 수 있어 대용량 에러 트래픽 처리 시 인프라가 안정적입니다.
  - 네트워크 단절 및 불안정 환경에서도 클라이언트 재구동 로직이 아주 단순합니다.
- **단점**:
  - 에러가 발생한 순간부터 대시보드에 집계되기까지 최대 5초의 시간 지연(Lag)이 발생할 수 있습니다 (이는 실무 요구 명세 범위 내에 속함).
