# ADR-005: SSE 좀비 연결 감지 방식 — `select-default` 논블로킹 전송 채택

> 작성일: 2026-07-15  
> 상태: 결정됨 (Decided)

## 컨텍스트

SSE(Server-Sent Events)는 서버 → 클라이언트 단방향 구조로, 클라이언트가 앱을 강제 종료하거나 네트워크가 단절되었을 때 서버는 즉각적으로 이를 감지할 수 없습니다. 이 때문에 실제로는 끊어졌지만 서버의 연결 맵(`Clients`)에는 여전히 남아 있는 **좀비 연결(Zombie Connection)** 이 시간이 지남에 따라 메모리에 누적됩니다.

`yoyaku_mate_server`에서 운영 중인 두 SSE 브로커(`Broker`, `WaitingUserBroker`)에 좀비 연결을 자동으로 감지하고 제거하는 Heartbeat 기능을 도입하기로 결정했으며, 구체적인 감지 방식에 대한 선택이 필요했습니다.

---

## 결정: `select-default` 논블로킹 send를 활용한 Heartbeat 방식 채택

30초 주기 백그라운드 고루틴(`startHeartbeat`)에서 전체 채널에 SSE 주석 이벤트(`:ping`)를 `select-default` 패턴으로 **논블로킹 전송**하고, 채널 전송이 블로킹되는 채널을 좀비로 판단하여 즉시 `close()` 후 맵에서 제거합니다.

```go
select {
case ch <- ":ping":  // 정상: 채널이 비어있어 즉시 전송 성공
default:             // 좀비: 채널이 블로킹 상태 → 즉시 제거
    delete(clients, ch)
    close(ch)
}
```

## 아키텍처별 비교 및 트레이드오프 (Trade-off Analysis)

### 대안 A: 양방향 Ping-Pong 응답 확인 방식

클라이언트가 ping 이벤트 수신 후 별도 HTTP API로 pong을 응답하고, 서버에서 타임아웃 내에 pong이 오지 않으면 좀비로 판단합니다.

* **장점**: 네트워크 레이어까지 포함한 end-to-end 연결 확인이 가능합니다.
* **단점**:
  * SSE는 단방향 프로토콜이므로 pong 수신을 위한 **별도 HTTP 엔드포인트**가 추가로 필요합니다.
  * 클라이언트 SDK(Flutter) 변경이 수반되며, pong 미응답 타임아웃 로직까지 작성해야 해 **구현 복잡도가 급증**합니다.
  * 클라이언트 네트워크 지연(Jitter)으로 인한 **false-positive**(정상 연결인데 좀비로 오인) 위험이 있어, 타임아웃 기준값 튜닝이 추가로 요구됩니다.

### 대안 B: `select-default` 논블로킹 전송 — **[채택]**

Go 채널의 `select-default` 패턴으로 논블로킹 send를 시도하고, 채널이 블로킹 상태이면 수신 고루틴이 없는 것(= 연결 끊김)으로 판단합니다.

* **장점**:
  * **클라이언트 코드 무변경**: SSE 스펙상 `:ping`은 주석으로 취급되어 클라이언트(Flutter `EventSource`)가 이를 이벤트로 수신하지 않으므로 클라이언트 수정이 전혀 필요 없습니다.
  * **단순성**: 추가 엔드포인트, 타임아웃 로직, 별도 상태 맵 없이 기존 `select` 문 하나로 감지가 완결됩니다.
  * **즉각 제거**: 감지 즉시 `close()` 처리로 메모리가 동기적으로 회수됩니다.
  * **Go 언어 특성 활용**: `chan string`은 포인터이므로 맵 키로 사용 가능하며, Go의 `range` 맵 순회 중 `delete`는 언어 스펙상 안전합니다.
* **단점**:
  * 채널의 수신 고루틴이 단순히 느린 경우(버퍼 미처리)를 좀비로 오인할 가능성이 있으나, 채널이 블로킹 상태가 된다는 것 자체가 핸들러 고루틴이 채널을 더 이상 소비하지 않는다는 의미이므로 실제로는 문제가 없습니다.
  * 30초 주기 실행 중 쓰기 락(`Lock()`)을 보유한 채 채널 send를 수행하므로, 해당 시간 동안 `Broadcast()` 호출이 일시 블로킹됩니다. (현재 규모에서는 허용 가능)

## 추가 고려사항: `:ping` 이벤트를 선택한 이유

일반 문자열(`"ping"`)이 아닌 콜론으로 시작하는 SSE 주석(`:ping`)을 사용하는 이유는 다음과 같습니다.

* SSE 스펙(RFC 8895)에서 `:` 으로 시작하는 라인은 **주석**으로 정의되어 있습니다.
* 클라이언트의 `EventSource` 또는 `StreamController`는 주석 라인을 이벤트로 파싱하지 않으므로 핸들러 함수가 불필요하게 호출되지 않습니다.
* 동시에 TCP 레이어에서는 실제 데이터가 전송되므로 NAT, 로드밸런서, 리버스 프록시(Nginx)의 idle connection timeout을 방지하는 **keep-alive** 역할도 겸합니다.

---

## 영향 (Consequences)

* **긍정적 영향**:
  * 클라이언트 SDK 변경 없이 서버 측 단독으로 좀비 제거가 완결됩니다.
  * 최대 30초 이내에 모든 좀비 연결이 자동으로 메모리에서 회수됩니다.
  * `:ping` 이벤트가 keep-alive를 겸하므로, NAT 타임아웃으로 인한 연결 강제 종료를 방지합니다.
* **제약 및 주의사항**:
  * `pingAndClean()` 실행 중 쓰기 락을 보유하므로, 연결 수가 수천 단위로 증가할 경우 Heartbeat 실행 구간 동안 Broadcast latency spike가 발생할 수 있습니다. 이 경우 "좀비 목록 수집 → 락 해제 → close() 처리" 패턴으로 리팩토링을 검토해야 합니다.
