# ADR-004: 인메모리 슬라이딩 윈도우 및 일별 활성 사용자 컬렉션을 활용한 접속자 트래킹

> 작성일: 2026-07-14  
> 상태: 결정됨 (Decided)

## 컨텍스트

대기열 및 예약 관리 시스템(Rusui)의 관리자 백오피스(`yoyaku_mate_admin`)에 플랫폼 활성화 지표 모니터링을 위한 '활성 사용자(Active User)' 기능 개발이 필요합니다.  
이를 위해 다음 두 가지 아키텍처적 과제에 대한 의사결정이 필요했습니다.
1. **실시간 동시 접속자 수(Concurrent Active Users)**를 stateless한 HTTP 요청 환경에서 어떻게 성능 부하 없이 정확하게 추적할 것인가?
2. **MAU(지난 30일간 접속자 수)**를 집계해야 하나, 기존의 `request_logs` 컬렉션은 디스크 용량 및 비용 최소화를 위해 **3일(259,200초) 만료 TTL 인덱스**가 적용되어 있어 과거 30일의 데이터가 보존되지 않는 문제를 어떻게 해결할 것인가?

---

## 결정: 5분 슬라이딩 윈도우 및 31일 TTL의 `daily_active_users` 수집 방식 채택

인프라 리소스 소모를 최소화하고 구현의 신뢰성을 극대화하기 위해 다음 결정을 내렸습니다.
1. **동시 접속자 추적**: 미들웨어를 통과하는 요청의 Client IP와 Timestamp를 thread-safe한 인메모리 맵에 실시간 갱신하고, **최근 5분 이내 활동(API 요청) 이력이 있는 고유 식별자(IP) 수**를 카운트하는 **5분 인메모리 슬라이딩 윈도우(Sliding Window)** 방식을 채택합니다.
2. **DAU/MAU 집계**: 5초 간격으로 동작하는 백그라운드 배치 flush 워커에서 수집된 IP들을 중복 제거하여, **31일(2,678,400초) 만료 TTL 및 `date`+`client_ip` 복합 유니크 인덱스가 적용된 `daily_active_users` 컬렉션**에 비동기 벌크 업서트(Bulk Upsert)하여 누적하고 이를 통해 DAU 및 MAU를 집계합니다.
3. **데이터 시딩 배제**: 서버 구동 시 기존 `request_logs` 데이터로부터 `daily_active_users`를 마이그레이션(시딩)하는 로직은 불필요한 복잡성 증가 및 중복 연산 유발을 막기 위해 제외하며, 신규 지표 수집 시점(배포 이후)부터 자연스럽게 데이터가 점진적으로 쌓이도록 설계합니다.

---

## 아키텍처별 비교 및 트레이드오프 (Trade-off Analysis)

### 1. 실시간 동시 접속자 집계 방식 비교

#### 대안 A: 실시간 SSE/WebSocket 커넥션 풀 카운팅 (Method B)
* **장점**: 소켓 커넥션의 연결/해제를 완벽히 추적하므로, 현재 화면을 켜놓고 대기 중인 유저 수를 가장 즉각적이고 물리적으로 반영함.
* **단점 (오류 및 자원 한계)**:
  * 네트워크 순단, 앱 강제종료, 브라우저 크래시 등 비정상 종료 시 커넥션 해제 감지(`-1` 처리)가 누락되어 실제 접속자보다 카운트가 비정상적으로 많이 찍히는 **유령 유저 누적 버그**가 쉽게 발생함. 이를 방지하기 위해 별도의 하트비트(Ping-Pong) 개발이 강제되어 공수가 급증함.
  * 동시 접속자 수만큼 상시 TCP 연결(소켓)이 열려 있어야 하므로, 서버 포트 및 커넥션 오버헤드가 발생하여 서버 리소스 비용이 상승함.

#### 대안 B: 5분 활동 기준 인메모리 슬라이딩 윈도우 (Method A) - **[채택]**
* **장점**:
  * **버그 제로 및 간결함**: API가 호출될 때 타임스탬프를 맵에 덮어쓰기만 하면 되므로 연결 유실 감지 실패로 인한 버그가 구조적으로 차단됨.
  * **서버 부하 제로**: Stateless HTTP를 사용하므로 소켓 상시 유지 비용이 들지 않음.
  * **실제적 활성 지표**: 단순히 화면을 백그라운드에 방치한 유저가 아니라, 실제로 API 요청을 유발시키며 활발히 서비스를 조작하는 진성 유저만을 카운트하여 비즈니스 가치가 높은 지표를 제공함.
* **단점**: 실시간 연결 상태를 실시간(ms 단위)으로 반영하지 않고 최대 5분의 지연 오차가 발생함. (대시보드 분석 목적상 이 정도의 시간차는 의사결정에 전혀 영향을 주지 않으므로 수용 가능)

---

### 2. MAU 집계를 위한 데이터 저장 방식 비교

#### 대안 A: 기존 `request_logs` 보관 주기를 30일로 연장
* **장점**: 추가 컬렉션 없이 기존 쿼리만 확장하면 되므로 백엔드 모델 추가 공수가 없음.
* **단점 (디스크 용량 폭증)**: 모든 API의 상세 정보(Path, Method, Latency, IP 등)가 담긴 로우 로그를 30일간 보관하게 됨. 하루에 40,000건의 요청만 들어와도 30일이면 120만 건 이상의 로우 데이터가 쌓여 MongoDB Atlas의 스토리지 추가 과금 요인이 됨.

#### 대안 B: 일별 고유 유저 기록 컬렉션 `daily_active_users` 분리 - **[채택]**
* **장점**:
  * **극단적인 공간 절약**: 동일한 유저가 하루에 1,000번 API를 호출해도 DB에는 `[날짜 - IP]` 단 1행(Row)의 도큐먼트만 업서트되어 누적되므로 데이터 크기가 수백 분의 일 수준으로 줄어듦.
  * **쿼리 속도 최적화**: 30일치 데이터 크기 자체가 작고, 복합 유니크 인덱스가 이미 생성되어 있으므로 MAU 집계(`Distinct`) 연산 속도가 매우 빠름.
* **단점**: 신규 컬렉션 생성 및 축적 기간이 필요합니다. (배포 이후 실시간 수집 시점부터 데이터가 점진적으로 쌓이게 되며, 초기 구동 시점의 불필요한 시딩 연산 로직을 배제하여 코드 단순성을 유지합니다.)

---

## 영향 (Consequences)

- **단기적 영향**: 어드민 웹 구동이나 인프라 추가 가동 비용 없이, 정확한 유저 활성 지표(동시 접속자, DAU, MAU)를 안전하게 제공합니다.
- **장기적 영향**: 유저 유입이 늘어나도 쓰기 부하와 저장 용량이 선형적으로 통제 가능한 범위 내에 머물게 되므로, 인프라 비용 리스크 없이 안정적인 메트릭 트래킹이 지속됩니다.
