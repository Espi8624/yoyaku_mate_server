# ADR-005: SSEゾンビ接続検知方式 — `select-default`によるノンブロッキング送信の採用

> 作成日: 2026-07-15  
> ステータス: 決定済み (Decided)

## コンテキスト

SSE(Server-Sent Events)はサーバーからクライアントへの単方向通信の特性上、クライアントがアプリを強制終了したり、通信環境不良によってネットワークが断絶したりした際、サーバー側で即時検知できません。これにより、実際には切断されているにもかかわらず、サーバーの接続管理マップ(`Clients`)には残存し続ける **ゾンビ接続(Zombie Connection)** が発生し、時間の経過とともにメモリをリークさせる原因となります。

`yoyaku_mate_server` が運用する2種類のSSEブローカー（`Broker`, `WaitingUserBroker`）に対し、ゾンビ接続を自動検知して解放するHeartbeat機能を導入することを決定し、具体的な接続判定の仕組みを選択する必要がありました。

---

## 決定: `select-default` ノンブロッキング送信を利用したHeartbeat方式の採用

30秒周期で実行されるバックグラウンドゴルーチン（`startHeartbeat`）からすべてのチャネルに向けて、SSE仕様上のコメントイベント（`:ping`）を `select-default` 構文を用いた **ノンブロッキング送信** で送信します。チャネルがブロック状態になっているクライアントをゾンビ接続と見なし、即時に `close()` の上、接続マップから削除します。

```go
select {
case ch <- ":ping":  // 正常: チャネルが空いているため即座に送信完了
default:             // ゾンビ: チャネルがブロックされている → 即座に削除
    delete(clients, ch)
    close(ch)
}
```

## 選択肢の比較およびトレードオフ (Trade-off Analysis)

### 代案A: 双方向Ping-Pongの応答検証方式

クライアントがサーバーからのpingイベントを受信後、別途用意したHTTP API経由でpong応答を返し、サーバーがタイムアウト期間内に応答を受信できなかった場合にゾンビ接続と判定します。

* **メリット**: ネットワーク層を含めた完全なエンドツーエンドの導通が検証可能です。
* **デメリット**:
  * SSEが単方向プロトコルであるため、pongを受け取るためだけの **別HTTPエンドポイント** の構築が追加で必要になります。
  * クライアント側（Flutter）の実装変更も同時に発生し、pong応答やその再試行制御ロジックを実装しなければならず、**開発難易度が急増**します。
  * クライアント端末側のネットワーク遅延（Jitter）によって、正常な接続をゾンビとして誤判定する **false-positive** のリスクがあり、タイムアウトパラメータの微調整が難しくなります。

### 代案B: `select-default` ノンブロッキング送信方式 — **[採用]**

Goのチャネル制御である `select-default` 構文により、チャネルへの送信をブロックさせずに実行を試み、バッファが埋まってブロックされる場合は受信ゴルーチンが正常に機能していない（＝接続喪失）とみなします。

* **メリット**:
  * **クライアントの実装変更が不要**: SSEの仕様上、コロン（`:`）から始まるパケットはコメント行とみなされるため、クライアント（Flutter `EventSource`）のイベントハンドラを汚しません。そのためクライアントへの追加実装が完全に不要です。
  * **シンプルなアーキテクチャ**: 追加のエンドポイントやタイムアウトマップ、ステート管理が一切不要で、Goの `select` 文一つで実装が完結します。
  * **即時回収**: 検知と同時に `close()` するだけで、関連ゴルーチンが抜けメモリが同期的に解放されます。
  * **Goの言語特性の活用**: `chan string` はポインタ型であるため、そのままマップのキーとして比較可能（ユニーク性担保）であり、Goにおける `range` ループ中の `delete` も言語仕様上スレッドセーフ（パニックを起こさない）です。
* **デメリット**:
  * 受信ゴルーチンの処理能力が単に一時的に低下している（バッファ遅延）場合をゾンビと誤判定する可能性が論理的には存在しますが、チャネルがブロックされる段階で既に該当接続のハンドラが正しく動作していないことを示すため、実用上問題ありません。
  * 30秒周期のクリーンアップ時、書き込みロック（`Lock()`）を占有した状態でチャネル送信を行うため、その間 `Broadcast()` が一時的に待機状態になります。 (現在の接続規模ではパフォーマンス上の影響は限定的)

## 補足: `:ping` イベントを採用した理由

単なる文字列の `"ping"` ではなく、コロンで始まる `:ping` を採用した理由は以下の通りです。

* SSE仕様（RFC 8895）において、コロンで始まるパケットは **コメント行** として定義されています。
* クライアントの `EventSource` はこれを受信しても解析せず読み捨てるため、不要な画面描画やコールバックハンドラの誤動作を防ぎます。
* 一方で、TCPレイヤー上ではパケットが実際に流れるため、NATルーターやロードバランサー（Nginx）、クラウドインフラがコネクションを切断する要因となるアイドルタイムアウトを防ぐ **keep-alive** の役割を担います。

---

## 影響 (Consequences)

* **メリット**:
  * クライアント側の変更を行うことなく、サーバー単体で安全かつ迅速にメモリ管理が行われます。
  * 最大30秒以内の遅延で、無駄なゾンビ接続が完全にメモリから回収されます。
  * 定期的な `:ping` の発生により、キャリア回線などのファイアウォールによる意図しない切断を軽減します。
* **懸念事項**:
  * 接続数が数千から数万規模にスケールした場合、 `pingAndClean()` 内での排他制御の競合時間が長くなるリスクがあります。将来的に接続過負荷となった場合は、「ゾンビチャネルの退避処理」と「実際のclose/削除処理」のフェーズを分ける非同期化設計への移行が必要です。
