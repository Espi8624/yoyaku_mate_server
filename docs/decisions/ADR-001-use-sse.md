# SSE vs WebSocket — 技術選定根拠

> 作成日: 2026-07-10  
> 決定者: 開発チーム  
> ステータス: 確定

## コンテキスト

ポーリング（Polling）方式で実装されていた待機列リアルタイム検知ロジックのリプレイスにあたり、  
双方向通信（WebSocket）と単方向サーバーPush（SSE）のどちらの方式を選択するか検討しました。

---

## 検討対象

| 項目 | SSE | WebSocket |
|------|-----|-----------|
| 方向 | サーバー → クライアント (単方向) | 双方向 |
| プロトコル | HTTP/1.1 上で動作 | 独自のWSプロトコル |
| 自動再接続 | ブラウザ標準サポート | 手動での実装が必要 |
| Go標準ライブラリ | `net/http`のみで実装可能 | 外部ライブラリが必要 |
| Proxy / ファイアウォール | 互換性が高い | 一部の環境でブロックされる |
| 適したユースケース | サーバーがデータをPushする場合 | チャット、ゲームなどの双方向リアルタイム |

---

## 決定: SSEの採用

### 理由

1. **単方向Pushで十分**: 待機列の変更はサーバーからクライアントへ伝達するだけで十分です。クライアントからサーバーへ送信するデータはREST APIで対応できます。

2. **Go標準ライブラリで実装可能**: `net/http` + `http.Flusher`のみでSSEを実装できるため、外部のWebSocketライブラリへの依存関係を追加する必要がありません。

3. **自動再接続**: ブラウザの `EventSource` APIは、接続が切断された場合に自動的に再接続を試みます。WebSocketではこのロジックを自前で実装する必要があります。

4. **fly.ioインフラとの互換性**: 現在のデプロイ環境において、WebSocketのロングコネクション処理はSSEに比べて複雑です。

### トレードオフ (許容可能)

- SSEはHTTP/1.1基準でブラウザあたりドメイン別6接続制限がありますが、現在の利用パターンでは問題ありません。
- クライアントからサーバーへデータをPushする必要がある機能が追加された場合、REST APIまたはWebSocketの再検討が必要です。
