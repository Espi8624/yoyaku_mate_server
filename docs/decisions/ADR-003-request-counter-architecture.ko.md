# ADR-003: 자체 메트릭 수집 및 리퀘스트 카운터 아키텍처 채택

> 작성일: 2026-07-13  
> 상태: 제안됨 (Proposed)

## 컨텍스트

대기열 관리 시스템(Rusui)의 관리자 백오피스(`yoyaku_mate_admin`)에 실시간 리퀘스트 통계(과거 24H 요청 수, 성공률, Peak TPS) 및 실시간 API 리퀘스트 로그 목록을 제공하기 위한 '리퀘스트 카운터' 기능 개발이 필요합니다.  
이를 구현하기 위해 상용 APM(Datadog, New Relic)을 도입할지, Prometheus+Grafana 같은 전문 모니터링 인프라를 직접 구축할지, 혹은 모니터링 엔진 자체를 밑바닥부터 자체 개발할지, 아니면 Go 백엔드와 MongoDB를 활용해 자체적으로 모니터링 기능을 구현할 것인지에 대한 의사결정이 필요했습니다.

---

## 결정: 인메모리 버퍼(배치 워커) 및 MongoDB TTL을 활용한 자체 수집 방식 채택

초기 서비스의 비용 효율성(Cost Efficiency)과 어드민 시스템의 단일 통합성(Single Dashboard Integration)을 확보하기 위해, 별도의 모니터링 시스템을 연동하는 대신 **Go 인메모리 버퍼링 기반 비동기 배치 적재 및 MongoDB TTL(Time-To-Live) 컬렉션을 이용한 자체 메트릭 수집 방식**을 채택합니다.

---

## 아키텍처별 비교 및 트레이드오프 (Trade-off Analysis)

### 1. 상용 APM 도구 (Datadog, New Relic 등) 도입
- **장점**: 별도 개발 없이 라이브러리 추가만으로 고도화된 시각화 대시보드와 병목 지점 실시간 모니터링 제공.
- **단점**: 도입 및 유지 비용이 스타트업 규모에서 매우 비쌈. 트래픽 유입량에 비례해 라이선스 비용이 크게 올라가므로 비용 통제가 불가능함.

### 2. 오픈소스 모니터링 솔루션 (Prometheus + Grafana) 직접 구축 (자가 호스팅)
- **장점**: 시계열 데이터 저장에 최적화(TSDB)되어 대규모 트래픽 수집 시 서버/DB 리소스 점유가 가장 적음. 오픈소스이므로 소프트웨어 라이선스 비용이 없음.
- **단점 (서버 비용 및 오버헤드)**:
  - **인프라 유지 비용 상승**: 현재 `fly.io`를 사용 중인데, 이를 가동하려면 최소 2개의 추가 컨테이너 인스턴스(Prometheus용 및 Grafana용)를 띄워야 하므로 호스팅 비용이 즉시 늘어남.
  - Grafana 대시보드용 별도 웹 접속 및 로그인 관리가 필요하여 어드민 통합성이 떨어짐.
  - 수치 지표(Metric)만 누적할 수 있어, 개별 요청의 상세 정보(예: 특정 IP의 실패 로그 내역 등)를 조회할 수 없음.

### 3. 전문 모니터링 엔진 자체를 밑바닥부터 직접 개발 (Go + 전용 TSDB 구축)
- **장점**: 우리 서비스에 100% 최적화된 독립 수집 시스템을 가질 수 있음.
- **단점 (인프라 및 기회비용)**:
  - 본 비즈니스 서비스(대기열/예약 관리) 개발 범위를 크게 벗어난 **과도한 엔지니어링(Over-engineering)**임.
  - 모니터링 엔진의 최적화와 저장소 유지 관리에 필요한 컴퓨팅 리소스 소모 및 별도 모니터링 데몬 기동으로 인해 서버 리소스 낭비 및 **간접 서버 비용**이 증가함.

### 4. 자체 경량 모니터링 (인메모리 버퍼 + MongoDB) - **[채택]**
- **장점**:
  - **서버 비용 0원**: 추가 서버 기동이나 유료 라이브러리 라이선스 구매 없이 기존 인프라(Go API 서버 + MongoDB Atlas 프리/기본 티어)를 그대로 공유하여 추가 비용이 전혀 없음.
  - **어드민 통합성**: 기존 리액트 어드민 웹 시스템 내부 대시보드 화면에 유기적으로 통합 렌더링 가능.
  - **Raw 로그 제공**: 특정 API 요청의 응답 속도 및 Client IP 정보를 Raw 로그 형태로 직접 조회하여 빠른 디버깅 지원 가능.
  - **애플리케이션 영향 최소화**: 5초 주기 백그라운드 고루틴(Bulk Insert) 처리를 통해 API 요청 처리 흐름에 주는 부하를 격리함.
- **단점 (잠재적 리스크)**:
  - **대용량 트래픽 시 스토리지 비용 상승**: 모든 API 리퀘스트 로그를 MongoDB에 적재하므로 트래픽이 많을 때 디스크 사용량이 누적되어 MongoDB Atlas의 스토리지 추가 비용이 발생할 수 있음.
  - **상쇄 조치**: `request_logs` 컬렉션에 **3일(259,200초) TTL 인덱스**를 지정하여 오래된 데이터가 자동으로 삭제되도록 설정, 디스크 점유 용량을 일정 수준 이하로 통제함으로써 스토리지 비용 상승 리스크를 원천 차단함.

---

## 영향 (Consequences)

- **단기적 영향**: 인프라 비용 추가 없이 간결하고 직관적인 API 모니터링 및 디버깅을 신속히 구현할 수 있습니다.
- **장기적 영향**: 향후 가입 점포 수 급증 및 사용자 트래픽 폭증 시(초당 수천 Req 돌파), MongoDB 쓰기 부하 경감을 위해 메트릭 데이터를 시계열 DB(Prometheus 등)나 별도 APM 도구로 마이그레이션해야 할 수 있습니다. 현재 구조는 과도기 아키텍처로서 가장 실용적인 대안입니다.
