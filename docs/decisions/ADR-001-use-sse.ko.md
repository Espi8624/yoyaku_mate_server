# SSE vs WebSocket — 기술 선택 근거

> 작성일: 2026-07-10  
> 결정자: 개발팀  
> 상태: 확정

## 컨텍스트

폴링(Polling) 방식으로 구현된 대기열 실시간 감지 로직을 교체하면서,  
양방향 통신(WebSocket)과 단방향 서버 Push(SSE) 중 어떤 방식을 선택할지 검토했습니다.

---

## 검토 대상

| 항목 | SSE | WebSocket |
|------|-----|-----------|
| 방향 | 서버 → 클라이언트 (단방향) | 양방향 |
| 프로토콜 | HTTP/1.1 위에서 동작 | 별도 WS 프로토콜 |
| 자동 재연결 | 브라우저 기본 지원 | 직접 구현 필요 |
| Go 표준 라이브러리 | `net/http`만으로 구현 가능 | 외부 라이브러리 필요 |
| Proxy / 방화벽 | 호환성 높음 | 일부 환경에서 차단 |
| 적합한 use case | 서버가 데이터를 Push하는 경우 | 채팅, 게임 등 양방향 실시간 |

---

## 결정: SSE 채택

### 이유

1. **단방향 Push로 충분**: 대기열 변경은 서버에서 클라이언트로만 전달되면 됩니다. 클라이언트가 서버로 보내는 데이터는 REST API로 충분합니다.

2. **Go 표준 라이브러리로 구현 가능**: `net/http` + `http.Flusher`만으로 SSE를 구현할 수 있어, 외부 WebSocket 라이브러리 의존성을 추가하지 않아도 됩니다.

3. **자동 재연결**: 브라우저의 `EventSource` API는 연결이 끊기면 자동으로 재연결을 시도합니다. WebSocket은 직접 구현해야 합니다.

4. **fly.io 인프라 호환성**: 현재 배포 환경에서 WebSocket의 롱 커넥션 처리가 SSE보다 복잡합니다.

### 트레이드오프 (수용 가능)

- SSE는 HTTP/1.1 기준 브라우저당 도메인별 6개 연결 제한이 있으나, 현재 사용 패턴에서는 문제 없음
- 클라이언트에서 서버로 데이터를 Push해야 하는 기능이 추가되면 REST API 또는 WebSocket 추가 검토 필요
