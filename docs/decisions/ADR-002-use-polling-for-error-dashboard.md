# ADR-002: エラーダッシュボードにおけるHTTPポーリングの採用

> 作成日: 2026-07-11  
> ステータス: 承認済み (Accepted)

## コンテキスト

待機列管理システム(Rusui)の管理者用バックオフィスにおいて、サーバー内で発生した各種エラー（4xx/5xx、DBエラー、SSE切断）をリアルタイムで監視・集計するエラーダッシュボード機能を実装する必要があります。  
現在、待機列の顧客通知機能には **SSE (Server-Sent Events)** 永続接続が採用されていますが、エラーの監視画面において、同様にSSE/WebSocketといった永続接続（リアルタイムプッシュ）を使用するべきか、あるいは定期的な **HTTPポーリング (Polling)** を使用するべきか意思決定が必要でした。

---

## 決定: 5秒周期のHTTPポーリングの採用

エラー監視システムとしての堅牢性とリソース最適化を考慮し、エラーダッシュボードのデータ取得には **5秒間隔のHTTPポーリング(Polling)方式** を採用します。

### 採用理由

1. **リアルタイム性の要求レベルの違い**
   - 待機列通知（SSE）は顧客の入場に関わるため「1秒未満の即時性」が必須ですが、管理者用のエラーダッシュボードは発生推移や累積統計を把握することが目的であるため、5秒程度の遅延は許容可能であり、ポーリングで十分にリアルタイム性を満たすことができます。
2. **接続維持コストとサーバーリソースの最小化**
   - SSEなどの常時接続方式は、ブラウザがダッシュボードを開いている間、サーバー側のソケットおよびコルーチン(Goroutine)リソースを永続的に消費します。
   - 一方、ポーリングは5秒ごとに瞬時のREST APIリクエストを送信して即座にコネクションを閉じるため、無駄な接続保持コストが発生しません。管理者ブラウザのタブが閉じられれば自動的にリクエストも停止します。
3. **障害発生時の自己復旧力 (Resiliency) の最大化**
   - モニタリングシステムは、メインサーバーが高負荷やネットワーク障害などの深刻なエラー状態に陥ったときこそ、最も確実に動作し続ける必要があります。
   - SSEのような常時接続は、接続が不安定な際の再接続ロジック（Backoffなど）の制御が複雑になりがちですが、HTTPポーリングは「リクエストに失敗しても次の5秒後に単にリトライする」だけであるため、極めてシンプルかつ自己復旧力が高い設計になります。

---

## 影響 (Consequences)

- **メリット**: 
  - サーバーのコネクションプール負荷を大幅に軽減できます。
  - 実装コードが極めて単純化され、ネットワーク切断に対する強度が上がります。
- **デメリット**: 
  - エラーの発生から最大で5秒間のタイムラグがダッシュボード上で発生します（要件上許容内）。
