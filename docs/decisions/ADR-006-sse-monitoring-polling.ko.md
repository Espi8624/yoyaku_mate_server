# ADR-006: SSE 모니터링 대시보드의 통신 격리 및 HTTP 폴링 방식 채택

> 작성일: 2026-07-15  
> 상태: 결정됨 (Decided)

## 컨텍스트

관리자용 SSE 상태 모니터링 대시보드(`/sse-status`)를 설계하면서, 실시간 연결 통계를 화면에 가져오는 인터페이스에 대한 논의가 있었습니다.

메인 앱의 실시간 기능이 SSE(Server-Sent Events)로 구축되어 있으므로 모니터링도 SSE 스트림을 사용하는 것이 타당해 보일 수 있으나, 이 경우 **동일한 브로커 시스템 상의 결함이 모니터링 도구 자체의 단절로 이어지는 전염 현상**이 우려되었습니다.

이에 따라 감시 대상(SSE 브로커)과 모니터링 수집 경로를 완전히 격리하는 설계가 요구되었습니다.

---

## 결정: 모니터링 독립성 확보를 위해 HTTP 폴링 방식 채택

SSE 스트리밍 대신, 5초 주기의 REST API 기반 **단기 HTTP 폴링(Polling)** 방식을 채택하여 모니터링 채널을 격리합니다.

### 채택 이유

1. **장애 결함 격리 (Fault Isolation)**:
   * SSE 스트리밍 방식을 채택하면 Admin의 모니터링 세션도 서버 내부의 SSE 브로커 자원(Mutex Lock, Goroutine)을 사용하게 됩니다.
   * 만약 서버 내 SSE 브로커의 Mutex 데드락이나 임계 오버헤드로 실시간 채널 전체가 마비되는 최악의 장애 상황이 발생할 경우, **SSE로 동작하는 모니터링 채널도 동시에 먹통이 됩니다.**
   * 무상태(Stateless) HTTP REST API로 통신 채널을 분리해 두면, SSE 브로커에 락이 걸리거나 통신이 마비되더라도 모니터링 도구는 계속 동작하여 관리자에게 "SSE 브로커가 현재 IDLE이거나 응답 불능"이라는 장애 지표를 정상적으로 시각화해 줄 수 있습니다.

2. **단순한 자기회복(Self-healing)**:
   * 영속 스트림 연결은 네트워크 순단 또는 프록시 타임아웃 발생 시, 정상 연결 상태 복원을 위해 프론트엔드에서 수동 재연결 및 지수 백오프(Exponential Backoff)를 정밀하게 제어해야 합니다.
   * HTTP 폴링은 매 회의 요청이 독립적이므로 특별한 예외 처리 없이 "실패하면 다음 5초 후에 자연스럽게 재시도"하는 단순 구조만으로 완벽한 자기회복 신뢰성을 지닙니다.

---

## 영향 (Consequences)

* **장점**:
  * SSE 장애 상황에서도 대시보드 채널이 차단되지 않아 시스템 가시성이 극대화됩니다.
  * 서버 메트릭 조회가 데이터베이스 조회 없이 인메모리(Memory map) 참조만으로 즉시 응답하므로, 5초 주기 폴링으로 인한 백엔드 CPU/네트워크 부하가 거의 없습니다.
* **단점**:
  * 연결 수 변동이나 좀비 제거가 발생하는 시점부터 대시보드에 반영되기까지 최대 5초의 지연(Time Lag)이 발생합니다. 이는 실시간 모니터링 목적상 운영 의사결정에 영향을 주지 않으므로 수용합니다.
