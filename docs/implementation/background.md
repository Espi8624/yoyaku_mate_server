# プロジェクト背景および設計意図

> 作成日: 2026-07-10

## 解決しようとした課題

### 1. リアルタイムデータ同期の遅延
数百人規模の待機顧客と店舗スタッフが待機列の状態変化（登録、呼び出し、完了、キャンセル）を遅延なく同期する必要がありますが、従来のポーリング方式はサーバーリソースを過剰に消費し、リアルタイム性に欠けていました。

### 2. ネットワークの不安定さによる重複リクエスト
モバイル環境において接続が一時的に切断されたり遅延したりする際、クライアントの再試行によって同一の待機登録が重複して生成されたり、順序の整合性が崩れたりする可能性がありました。

### 3. 不正な過剰リクエストおよびセキュリティの脅威
待機登録時のマクロ攻撃や非認可API呼び出しによりサーバーの負荷が加重される可能性があり、また非認可ユーザーによるデータベース直接操作の危険が存在していました。

### 4. 非定型大容量データの管理
店舗の詳細設定、頻繁に変更されるメニュー構造、日別の待機データ統計化、事業者登録証・メニュー画像の大容量アップロードを高速で処理しながら、データの整合性を維持する必要がありました。

---

## 解決方法

### 1. GoroutineベースのSSE Brokerの構築
重いWebSocketの代わりに、GoチャネルとGoroutineベースの軽量なSSEストリーミングエンドポイントを実装し、待機列の状態変更を検知した際に接続中のクライアントへ1秒以内に単方向Pushストリーミングします。

→ [SSE実装詳細](./sse.md)

### 2. クライアントベースのUUID + べき等性検証設計
クライアントがリクエスト送信時に `waiting_id` を直接生成して送信し、サーバーでDBと対照して重複している場合は既存のデータをそのまま返却することで、重複登録を根本的に防止します。

→ [冪等性実装詳細](./idempotency.md)

### 3. ミドルウェアベースのセキュリティレイヤー
`didip/tollbooth` によりIPごとの秒間リクエスト数を制限し、Firebase Admin SDKによってすべての主要なバックオフィスAPIのJWTトークンの有効性をサーバー側で検証します。

### 4. MongoDB Atlas + Cloudflare R2の組み合わせ
柔軟な非定型データ処理のためにMongoDB NoSQLを採用し、画像などの重いファイルアセットのアップロードはS3 API互換ストレージであるCloudflare R2に委任することで、DBサーバーの負荷を分散します。
