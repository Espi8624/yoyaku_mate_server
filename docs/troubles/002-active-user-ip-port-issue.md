# 002: リアルタイム接続者の重複カウント防止 (IPポートおよびIPv6の単一化)

> 作成日: 2026-07-14  
> 状態: 解決済み (Resolved)  
> 関連ファイル: [`metrics/middleware.go`](../../metrics/middleware.go)

---

## 現象 (Symptom)

管理者バックオフィス（`yoyaku_mate_admin`）のアクティブユーザーダッシュボードにおいて、以下のような異常なカウント現象が確認されました。
1. **継続増加現象**: 同一のデバイス/ブラウザでダッシュボードを開いている状態にもかかわらず、フロントエンドが5秒ごとにAPIポーリング要求を送信するたびに、`CURRENT ACTIVE USERS` の数値が継続して1ずつ増加しました。
2. **複数検知現象**: ローカル環境で明らかに1人の開発者のみが接続している状態であるにもかかわらず、接続者数が `2` と表示されました。

---

## 原因分析 (Root Cause)

### 1. 一時ポート（Port）番号が含まれる識別子キーの問題
Go バックエンドの `r.RemoteAddr` は、単純なIPアドレスではなく `IP:Port` 形式の文字列（例: `127.0.0.1:54321`）を返します。
ブラウザがHTTPポーリングを行うたびに、クライアントポートは**エフェメラルポート（Ephemeral Port - 一時ポート）**の規則に従ってランダムな値に毎回切り替わります。
ポート番号を切り離さずにそのままユーザー識別子キーとしてマップに登録していたため、同じデバイスから送信されるリクエストであるにもかかわらず、サーバー側は毎回新しいユニークユーザーが流入したと判定し、カウントが上昇し続けました。

### 2. IPv4とIPv6ループバックアドレスの二重認識問題
ローカル起動（localhost）環境において、ブラウザがドメイン名前解決を行う際に、IPv4のループバックアドレスである `127.0.0.1` と、IPv6のループバックアドレスである `::1` を交互に使用してリクエストを送信していました。
サーバー側のマップ上では、これら二つのアドレスは異なる文字列として扱われるため、個別のユニークユーザーとして集計され、最終的な結果が `2` と表示されました。

---

## 解決策 (Solution)

### 1. `net.SplitHostPort` を用いたポート番号の分離
ミドルウェアでIPアドレスを取得する際、Go 標準ライブラリの `net.SplitHostPort` を利用し、ポート部分を除外して**純粋なIPアドレス部分のみを動的にスライシング**するように変更しました。

### 2. IPv6ループバックアドレスの単一化 (Normalization)
ローカル開発環境およびデュアルスタックネットワーク環境における接続者カウントの整合性を保つため、収集されたIPアドレスが `::1` である場合は、自動的に `127.0.0.1` にマッピング（統一）するクレンジングフィルターを追加しました。

```go
// MetricsMiddleware 内 IP クレンジング実装部
clientIP := r.Header.Get("X-Forwarded-For")
if clientIP == "" {
    ip, _, err := net.SplitHostPort(r.RemoteAddr)
    if err == nil {
        clientIP = ip
    } else {
        clientIP = r.RemoteAddr
    }
} else {
    ips := strings.Split(clientIP, ",")
    clientIP = strings.TrimSpace(ips[0])
}

// IPv6ループバックをIPv4ループバックへ統一して重複防止
if clientIP == "::1" {
    clientIP = "127.0.0.1"
}
```

---

## 結果と整理 (Consequences)
- **精密な識別**: ブラウザの継続的なポーリング要求に対しても同一IPとして識別され、重複カウントの問題が完全に解決されました。
- **マルチドメイン対応**: ローカル接続経路（IPv4/IPv6）に関わらずループバックアドレスが統合され、正確に `1` 名として正しく追跡されます。
- **教訓**: ソケットやネットワークパケットに基づく接続者トラッキングでは、ポート番号の切り離しとループバック（開発環境）の正規化処理を基本的な例外処理として考慮する必要があります。
