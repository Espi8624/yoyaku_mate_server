# 機能仕様書: SSE状態監視 (SSE Status Monitoring)

本ドキュメントは、 `yoyaku_mate_server` Goバックエンドサーバーが提供するSSEブローカー接続状況のリアルタイム監視機能の仕様を説明します。

> 作成日: 2026-07-15  
> 関連ドキュメント: [実装詳細書: SSE状態監視](../implementation/sse-monitoring.md), [実装詳細書: SSEリアルタイム接続アーキテクチャ](./sse.md)

---

## 1. 概要 (Overview)

`yoyaku_mate_server` は2種類のSSEブローカーを運用しています。

| ブローカー | 役割 | 購読キー |
|--------|------|---------|
| `Broker` | 店舗全体の待ちリスト変更事項のリアルタイム伝達 (スタッフ/店主対象) | `storeID` |
| `WaitingUserBroker` | 個別待ち顧客のステータス変更のリアルタイム伝達 (顧客対象) | `storeID:waitingID` |

SSEはサーバーからクライアントへの単方向接続であるため、クライアントの接続解除シグナルをサーバーが即座に検知できません。この特性により、接続が維持されているように見えても実際には切断されている **ゾンビ接続(Zombie Connection)** がメモリ内に累積する可能性があります。

本機能は、これら2つのブローカーの接続状況をリアルタイムで集計してAdminダッシュボードに公開し、**30秒周期のHeartbeatゴルーチン**によってゾンビ接続を自動的に削除します。

---

## 2. 核心指標 (Key Metrics)

Adminダッシュボードの `/sse-status` ページで提供され、5秒周期のHTTPポーリングによって更新されます。

### 2.1 要約指標カード (Metrics Cards)

* **TOTAL CONNECTIONS (全体有効接続数)**
  * **説明**: 2つのブローカーの有効なチャネル数を合算した現在の総SSE接続数です。
  * **算出基準**: `Broker.TotalConnections + WaitingUserBroker.TotalConnections`

* **CONNECTION HEALTH (接続健全性状態)**
  * **説明**: 全体接続数を基準に判定したブローカーの動作状態です。
  * **反映基準**: `TotalConnections > 0` → `HEALTHY` / `0` → `IDLE`

* **STORE BROKER (店舗待ちリストブローカー)**
  * **説明**: 店舗待ちリストSSEを購読中の店舗数、接続数、および平均維持時間です。
  * **反映基準**: `Broker.GetStats()` 呼び出し結果 (インメモリ参照、DBアクセスなし)

* **USER BROKER (個別待ち顧客ブローカー)**
  * **説明**: 個別待ち顧客SSEを購読中のユーザーキー数、接続数、および平均維持時間です。
  * **反映基準**: `WaitingUserBroker.GetStats()` 呼び出し結果 (インメモリ参照、DBアクセスなし)

## 3. Heartbeat — ゾンビ接続の自動削除

### 3.1 動作方式

1. サーバーの初期化時に各ブローカーのシングルトンで `startHeartbeat()` ゴルーチンが自動実行されます。
2. **30秒周期**で `pingAndClean()` が全体のチャネルにSSEコメントイベント（`:ping`）をノンブロッキング送信（`select-default`）します。
3. チャネルがブロック状態（バッファが一杯、または受信ゴルーチンの終了） → ゾンビと判定 → `close()` およびマップから即時削除します。
4. `:ping` はSSE仕様のコメント形式であるため、正常なクライアントはこれをイベントとして受信しません。

### 3.2 検知可能なケース

* クラアントアプリの強制終了 (defer未実行)
* ネットワーク切断後の再接続なし放置
* アプリのバックグラウンド遷移による接続維持中断

---

## 4. 今後の高度化ロードマップ (Roadmap)

### 4.1 第2段階: 接続履歴分析

* **ピーク接続数の記録**: サーバー起動後の最大同時接続数をインメモリで記録し、ダッシュボードに表示します。
* **チャネルバッファサイズの調整**: ゾンビ検知の感度を高めるため、チャネルバッファサイズ設定を環境変数で管理できるようにします。

### 4.2 第3段階: 運用の高度化

* **強制接続解除API**: Admin UIから特定の `storeID` または `waitingID` 購読を強制終了するエンドポイントを追加します。
* **Heartbeat周期のRuntime調整**: 環境変数またはRemote Configを通じて、Heartbeatの実行周期をサーバー再起動なしで変更できるようにします。
